

# 概念

nmem 是 AI 代理的分层记忆系统。本页解释了心理模型：每一层的作用、条目如何在各层之间流动，以及巩固引擎的工作原理。

## 6 层级体系

将 nmem 想象成人类记忆：

```
工作记忆  →  日志  →  LTM  →  共享知识
(正在做的事情)   (发生了什么)   (已知内容)   (人人皆知)

实体记忆：每人/对象的笔记本
策略记忆：所有人必须遵循的规则
```

### 第 1 层：工作记忆

**是什么：** 当前会话的临时键值槽。像是一块草稿纸。

**用途：** 当前任务、活跃上下文、草稿笔记。会话结束时清除。

**示例：**
```python
await mem.working.set("session-1", "agent-1", "current_task", "调试支付超时")
```

**生命周期：** 仅限会话。永不提升。

### 第 2 层：日志

**是什么：** 30 天的活动日志。每个重要的观察、决定和结果都会在此记录。

**用途：** 会话摘要、做出的决定、学到的教训、注意到的事物。这是主要的写入目标：大多数代理交互应创建日志条目。

**示例：**
```python
await mem.journal.add(
    agent_id="support",
    entry_type="lesson_learned",
    title="超过 100 美元的退款需要经理批准",
    content="客户请求 150 美元退款。发现这需要...",
    importance=7,
)
```

**生命周期：** 30 天后过期。高重要性条目（≥7）自动提升到 LTM。被频繁访问的条目（≥5 次访问）也会提升。

**条目类型：** `observation`、`decision`、`lesson_learned`、`session_summary`、`investigation`、`interaction`、`analysis`

### 第 3 层：长期记忆（LTM）

**是什么：** 每个代理的永久知识。已证明有价值的事实、程序和教训。

**用途：** 应该永久保留的知识：程序（“如何部署”）、事实（“团队站会在 9:30”）、教训（“永远不要在周五部署”）。

**示例：**
```python
await mem.ltm.save(
    agent_id="engineering",
    category="procedure",
    key="deploy_checklist",
    content="1. 运行迁移，2. 种子缓存，3. 验证健康检查",
    importance=8,
)
```

**生命周期：** 永久。**显著性**（条目对当前推理的影响强度）如果 90 天以上未被访问会衰减——知识不会被删除，只是排名降低，直到某些东西重新激活它。当 ≥2 个不同代理访问时，自动提升到共享知识。

> **显著性 vs 依据**：显著性不是真实性度量。显著性低的条目可能仍然真实——只是冷了。对于真实性/依据的确定性，请参见 `record_type`（`evidence` / `judgment` / `fact`）和 `grounding`（`source_material` / `confirmed` / `inferred` / `disputed`）。实体记忆使用单独的 `confidence` 字段，具有严格的依据语义。

**类别：** `fact`、`procedure`、`lesson`、`pattern`、`policy`、`contact`、`troubleshooting`、`architecture`

### 第 4 层：共享知识（社会学习）

**是什么：** 跨代理的规范事实。对所有代理可见。

**用途：** 公司范围的知识：政策、供应商联系人、升级程序、影响所有人的架构决策。

这一层是 nmem 中 **社会学习** 的核心。代理不仅维护私有记忆，还贡献于共享知识库，使每个代理随着时间推移变得更聪明。学习循环：

1. 代理观察到某事 → 写入日志
2. 如果足够重要 → 提升到私有 LTM
3. 如果其他代理开始访问它 → 自动提升到共享知识
4. 每个代理的提示注入现在都包含这些知识

结果：当一个代理学到一个教训，所有代理都受益——无需显式编程或代理之间的消息传递。

**示例：**
```python
await mem.shared.save(
    agent_id="system",
    category="policy",
    key="refund_policy",
    content="30 天退款窗口。超过 100 美元需要经理批准。",
    importance=9,
)
```

**生命周期：** 永久。带有变更日志的版本。创建的三种路径：
1. **直接保存**：代理显式保存共享知识
2. **提升**：LTM 条目被 ≥2 个不同代理访问，证明跨代理相关性
3. **夜间合成**：巩固引擎在所有代理的日志条目中发现跨切面模式

### 第 5 层：实体记忆

**是什么：** 每个业务对象（客户、错误、部署等）的协作笔记本。多个代理读写。

**用途：** 构建特定实体的档案。关于客户的支援笔记、关于错误的工程笔记、关于潜在客户的销售笔记，都在一个地方。

**示例：**
```python
await mem.entity.save(
    entity_type="customer",
    entity_id="cust-jane-doe",
    entity_name="Jane Doe",
    agent_id="support",
    content="报告订单 #5678 的双重收费。退款已启动。",
    record_type="evidence",
    confidence=1.0,
    grounding="source_material",
)
```

**记录类型：**
- `evidence`：从原始材料中观察到的事实（最高置信度）
- `judgment`：推断出的结论（强制置信度 <1.0）
- `task`：要执行的操作
- `summary`：所有记录的汇总

**依据等级：**
- `source_material`：直接来自原始来源
- `inferred`：通过推理得出
- `confirmed`：通过多个来源验证
- `disputed`：被其他记录反驳

### 第 6 层：策略记忆

**是什么：** 带有写入者/提议者权限的治理规则。

**用途：** 系统必须遵循的硬性规则：访问控制、安全防护、操作约束。只有指定的“写入者”代理可以创建活动策略；其他代理可以提议。

**示例：**
```python
await mem.policy.save(
    scope="global",
    category="safety",
    key="no_pii_in_logs",
    content="从不记录 PII（电子邮件、电话、SSN）。在记录前屏蔽所有 PII。",
    agent_id="system",
)
```

## 条目如何在各层之间流动

```
         ┌──────────────────────────────────────────────┐
         │           工作记忆（会话）            │
         └─────────────────────┬────────────────────────┘
                               │ 会话结束
                               ▼
         ┌──────────────────────────────────────────────┐
         │              日志（30 天）                │
         │                                              │
         │ 重要性 ≥ 7  OR  访问次数 ≥ 5        │
         └─────────────────────┬────────────────────────┘
                               │ 巩固提升
                               ▼
         ┌──────────────────────────────────────────────┐
         │           长期记忆（永久）        │
         │                                              │
         │  被 ≥ 2 个代理访问  AND  重要性 ≥ 8 │
         └─────────────────────┬────────────────────────┘
                               │ 基于访问的提升
                               ▼
         ┌──────────────────────────────────────────────┐
         │        共享知识（跨代理）         │
         └──────────────────────────────────────────────┘
```

关键原则：**条目通过自身价值获得提升**。没有任何条目是通过 LLM 对“普遍性”的猜测而被提升的。提升由作者设置的重要性分数和其它代理的访问模式驱动。

## 巩固引擎

巩固引擎在后台运行（默认每 6 小时一次），执行 10 个步骤：

| 步骤 | 做什么 | 为什么重要 |
|------|-------------|----------------|
| 1. 衰减过期 | 删除低重要性过期日志条目 | 防止无限制增长 |
| 2. 提升到 LTM | 将高重要性日志条目移动到永久存储 | 保存有价值的知识 |
| 3. 提升到共享 | 将跨代理 LTM 条目移动到共享知识 | 多个代理需要的知识成为规范 |
| 4. 去重 LTM | 聚合相似条目（余弦 >0.85），通过 LLM 合并 | 防止冗余知识 |
| 5. 自动重要性 | 使用启发式方法重新评分标记为 `auto_importance=True` 的条目 | 保持重要性与实际使用模式一致 |
| 6. 信念修订 | 使用依据等级 → 代理信任 → 新鲜度 → 重要性解决待定冲突 | 矛盾得到解决，失败者标记为过时 |
| 7. 显著性衰减 | 减少未被访问 90 天以上的条目的显著性 | 过时知识从当前推理中淡出（但不会被删除——只是排名较低） |
| 8. 自定义钩子 | 运行特定于应用程序的步骤 | 可扩展性 |
| 9. 知识链接 | 在相关条目之间建立关联链接（实体、标签、时间） | 通过相关条目实现搜索扩展 |
| 10. 好奇心衰减 | 减少旧的未解决好奇心信号的评分 | 防止信号疲劳 |

**微循环：** 当创建一个高重要性条目（≥7）时，巩固器立即唤醒进行快速提升。关键知识在几秒内到达 LTM。

**夜间合成：** 每天一次，引擎分析过去 24 小时的所有日志条目，使用 LLM 提取 2-3 个跨切面模式，并将其保存为共享知识。它还会追溯性地提升对发现模式有贡献的日志条目的重要性。

**回顾（梦境状态）：** 夜间合成完成后，引擎回顾过去的教训与新证据——就像大脑在睡眠中巩固一样。它会提取 `record_type` 与教训模式（默认：`lesson`、`lesson_learned`）匹配的 LTM 条目，这些条目是在 `lookback_days`（默认 14）内创建的，跳过在 `skip_if_validated_within_days`（默认 3）内验证过的条目。对于每个条目，它会在日志中搜索在教训之后创建的结果条目，然后要求 LLM 分类：`reinforces`（提升自动管理条目的显著性 + 重要性）、`contradicts`（标记 `grounding='disputed'`）、或 `neutral`（跳过，稍后重新审视）。每晚受 `max_llm_calls_per_run`（默认 5）限制。发现的结果以 `category="retrospective_synthesis"` 写入共享知识。

## 混合搜索

nmem 中的每个搜索结合两个信号：

- **向量相似度（60%）**：pgvector 余弦距离与嵌入
- **全文搜索（40%）**：PostgreSQL `tsvector` / `ts_rank_cd`

这意味着搜索“部署流程”会找到：
- 语义相似的条目（向量捕捉“部署程序”、“发布工作流”）
- 包含确切词语的条目（FTS 捕捉“部署”即使嵌入模型不将其排名最高）

60/40 的权重是根据 Spwig 炼油厂系统生产经验调整的。向量捕捉语义，FTS 捕捉精确术语。

## 上下文线程

nmem 自动使用余弦相似度（阈值：0.65）将相关条目聚类为“上下文线程”。当你添加一个关于“数据库备份开始”的日志条目，之后添加一个关于“数据库备份完成”的条目，它们会被分配到同一个线程。

这启用了未来功能，如线程范围搜索（“显示所有关于备份事件的内容”）和对话摘要。

## 提示注入

提示构建器为注入到代理系统提示中的记忆上下文进行组装：

```python
ctx = await mem.prompt.build(agent_id="support", query="支付问题")
system_prompt = f"您是一名支持代理。\n\n{ctx.full_injection}"
```

注入使用 **分层详细程度**：
- **策略**：全文（规则必须完整）
- **共享知识**：带有键的截断存根
- **LTM**：类别 + 键 + 截断内容
- **日志**：日期 + 类型 + 标题（使用 `memory_search` 获取详细信息）
- **工作记忆**：槽名 + 内容

这保持注入紧凑（约 300-500 个 token），同时让代理访问最相关的上下文。

## 令牌趋势

nmem 自动跟踪提示注入大小和 LLM 成本。每次调用 `mem.prompt.build()` 都会记录每部分的 token 估计值，巩固操作（压缩、合成、去重）会记录其 LLM 使用情况。通过 API（`GET /v1/token-trends`）或 CLI（`nmem token-trends`）查询趋势。

随着时间的推移，您应该看到每次提示构建的平均 token 数 **稳定或减少**，因为巩固引擎去重知识、压缩冗长条目，分层详细程度保持注入紧凑。如果每次调用的平均 token 数在增加，这表明巩固阈值可能需要调整。

## 多代理系统中的社会学习

在单代理设置中，nmem 是一个持久记忆，减少跨会话的重新调查。在 **多代理** 设置中，nmem 变成一个社会学习系统：

```
代理 A 观察 → 日志 → 提升到 LTM
                                    ↓
                          代理 B 访问它
                                    ↓
                   自动提升到共享知识
                                    ↓
                所有代理在提示注入中接收它
```

驱动这一过程的三个机制是：

1. **基于访问的提升**：当 ≥2 个不同代理访问同一个 LTM 条目时，系统识别为跨代理相关，并自动将其提升到共享知识。

2. **夜间合成**：巩固引擎分析 *所有* 代理的日志条目，提取单个代理无法看到的模式。支持代理的客户投诉 + 工程代理的部署日志 → 合成关于部署时间的见解。

3. **信念修订**：当代理意见不一致（矛盾知识）时，冲突检测系统标记矛盾，并使用依据等级、代理信任和新鲜度解决，而不是静默地用最后写入覆盖。

效果是累积的：每个代理的个人学习改善了集体知识库，而集体知识库使每个代理的会话上下文更丰富、更准确。
